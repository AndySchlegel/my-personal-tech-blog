Key Design Decisions

  • ALB target-type: ip — VPC CNI routet direkt zu Pod IPs
  • Spot Instances — t3.medium + t3a.medium (~70% günstiger)
  • nginx Reverse Proxy — ALB → nginx → /api/* → Backend
  • kubectl create secret --dry-run — echte Werte, nie hardcoded
  • Liveness + Readiness — HTTP /health (Backend), TCP + HTTP (Frontend)
  • DB Init als Job — idempotentes Schema + Seed gegen RDS
  • IRSA — Pod-Level IAM für Comprehend + Translate + Polly (3 Rollen)
  • HPA + Metrics-Server — CPU-basiertes Auto-Scaling (1-4 Backend, 1-3 Frontend)

Lessons Learned

42 dokumentierte Learnings — jedes mit konkretem Problem und Lösung

#8 IRSA statt Node IAM

IMDS auf EKS-Nodes blockiert — Pod-Level IAM via IRSA ist Pflicht. Ohne ServiceAccount-Annotation kein Zugriff auf Comprehend, Translate, Polly.

#17 Dynamic Terraform Outputs

Pipeline liest Infra-Werte aus Terraform State — kein manuelles Secret-Update nach Destroy+Apply.

#21 Circular Dependency Deadlock

OIDC/IAM Zirkularabhängigkeit. Fix: lifecycle ignore_changes + Wave 0 Pre-Step.

#42 Self-Hosted vs Managed Grafana

Self-hosted Grafana via Helm statt Amazon Managed Grafana (~$9/Monat). Gleiche Dashboards, $0 extra auf Spot Nodes. Managed lohnt erst bei 24/7 Teams.

#26 ALB Controller Orphans

K8s Controller erstellt ALBs außerhalb von Terraform. Cleanup BEFORE EKS Destroy nötig.

#30 Full Lifecycle Verified

Reproduzierbarkeit beweist man durch den vollen Zyklus. Jede Iteration deckt neue Edge Cases auf.

#24 Separate ACM-Zertifikate

CloudFront verlangt ACM-Cert in us-east-1, ALB in eu-central-1. Zwei separate Zertifikate für die gleiche Domain — globale vs. regionale Services leben in getrennten Welten.

#32 Dual-Track Hosting

EKS für Showcase ($143/Sprint), Lightsail für Dauerbetrieb ($5.50). Kubernetes-Expertise zeigt sich daran, wann man es NICHT nutzt.

#35 WICHTIG Comprehend: Deutsche Ironie

"Ganz schön überzogen dargestellt!" → POSITIVE 100%. Comprehend erkennt keinen deutschen Sarkasmus. Manuelle Moderation bleibt Pflicht.

#14 EKS Cluster Security Group

EKS erstellt eine eigene Cluster-SG — Custom Node-SG-Regeln greifen nicht. ALB-to-Pod Traffic muss auf die Cluster-SG, nicht auf die eigene. Gilt auch für den ALB Controller.

Warum EKS gewählt

  • Tiefes Verständnis für Container-Orchestrierung
  • Enterprise-relevante Architektur (Deployments, Services, Ingress, HPA)
  • IRSA: Pod-Level IAM statt Node-Level — Security Best Practice
  • ALB Controller: K8s Ingress erzeugt echte AWS ALBs automatisch
  • CKA-Vorbereitung als Lernziel

Kostenbewusstsein

  • EKS nur für Showcase/Demos — gezielt hoch- und runterfahren
  • Spot Instances (t3.medium + t3a.medium) — ~70% günstiger
  • Node Group auf 0 skalierbar wenn nicht aktiv
  • Lightsail als Dauerbetrieb-Alternative (~$5.50/Monat)
  • terraform destroy runterfahren wenn nicht entwickelt wird

"K8s und Lambda sind die zwei dominierenden Muster im Cloud-Bereich. Wer beides kennt hat einen klaren Vorteil. Für das Abschlussprojekt reicht eines — aber es lohnt sich, das andere danach selbst auszuprobieren."

— Aufgabenstellung, Seite 2

In-Memory Key-Value Store für Session-Caching, API-Response-Caching, Rate Limiting, Pub/Sub Messaging und Leaderboards. Sub-Millisekunden Antwortzeiten.

Warum hier nicht sinnvoll?

Translate-Cache liegt in PostgreSQL (post_translations), Polly-Audio in S3. Bei ~12 Posts und geringem Traffic würde Redis nur Komplexität hinzufügen ohne messbaren Vorteil.

Wann lohnt sich Redis?

Ab hunderten gleichzeitigen Usern, bei Session-basierter Auth, oder wenn DB-Queries zum Bottleneck werden. Dann cache.t3.micro (~$13/Monat) als ElastiCache.
